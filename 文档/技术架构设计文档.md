## 1. 技术栈选型

### 1.1 前端技术栈

## 4. 核心技术方案

### 4.1 用户认证方案（JWT）

#### 认证流程

```mermaid
sequenceDiagram
    participant Client as 前端
    participant Server as 后端

    Client->>Server: POST /api/auth/login (phone, password)
    Server->>Server: 验证用户名密码
    alt 验证成功
        Server-->>Client: 返回 { token, userInfo }
        Client->>Client: 存储token到localStorage
        Client->>Server: 后续请求携带Authorization: Bearer {token}
        Server->>Server: 解析JWT验证有效性
        Server-->>Client: 返回业务数据
    else 验证失败
        Server-->>Client: 返回401错误
    end
```

#### Token结构

| Token类型 | 有效期 | 存储位置 |
|-----------|--------|----------|
| Access Token | 24小时 | 前端localStorage |

#### JWT Payload示例

```json
{
  "userId": 10001,
  "username": "zhangsan",
  "role": "helper",
  "exp": 1700007200,
  "iat": 1700000000
}
```

### 4.2 权限控制（RBAC三角色隔离）

#### 角色定义

| 角色 | 标识 | 权限说明 |
|------|------|----------|
| 需求方 | requester | 发布任务、验收任务、评价任务 |
| 接单方 | helper | 浏览任务、接单、提交完成 |
| 管理员 | admin | 所有权限，包括审核、争议处理 |

#### 权限拦截流程

```mermaid
flowchart TD
    A[请求到达] --> B{Token验证}
    B -->|无效/过期| C[返回401]
    B -->|有效| D[解析用户角色]
    D --> E[检查接口权限配置]
    E --> F{角色匹配?}
    F -->|不匹配| G[返回403]
    F -->|匹配| H{资源归属验证}
    H -->|非所有者| I[返回403]
    H -->|所有者/公共资源| J[执行业务逻辑]
    J --> K[返回响应]
```

#### 权限注解示例

```java
// 需求方或管理员可访问
@RequiresRoles(value = {"requester", "admin"})
@PostMapping("/tasks")
public ResponseEntity createTask(...) { ... }

// 仅接单方可访问
@RequiresRole("helper")
@PostMapping("/tasks/{taskId}/accept")
public ResponseEntity acceptTask(...) { ... }
```

### 4.3 任务状态机方案

#### 状态流转图

```mermaid
stateDiagram-v2
    [*] --> PENDING : 发布任务
    PENDING --> ONGOING : 接单
    PENDING --> CANCELLED : 取消任务
    ONGOING --> PENDING_REVIEW : 提交完成
    ONGOING --> DISPUTED : 发起争议
    PENDING_REVIEW --> COMPLETED : 验收通过
    PENDING_REVIEW --> ONGOING : 验收拒绝
    PENDING_REVIEW --> DISPUTED : 发起争议
    DISPUTED --> COMPLETED : 管理员判决
    DISPUTED --> CANCELLED : 管理员判决
    COMPLETED --> [*]
    CANCELLED --> [*]
```

#### 状态枚举定义

| 状态 | 标识 | 说明 |
|------|------|------|
| 待接单 | PENDING | 任务已发布，等待接单 |
| 进行中 | ONGOING | 任务已被接单，正在执行 |
| 待验收 | PENDING_REVIEW | 接单方已提交完成，等待验收 |
| 已完成 | COMPLETED | 验收通过，任务完成 |
| 争议中 | DISPUTED | 存在争议，等待管理员处理 |
| 已取消 | CANCELLED | 任务已取消 |

### 4.4 资金冻结与事务方案

#### 资金流转流程

```mermaid
sequenceDiagram
    participant User as 用户
    participant TaskService as TaskService
    participant AccountService as AccountService
    participant DB as MySQL

    User->>TaskService: 发布任务(金额15.75)
    TaskService->>AccountService: freeze(userId, 15.75)
    
    AccountService->>DB: BEGIN TRANSACTION
    AccountService->>DB: UPDATE account SET balance = balance - 15.75 WHERE userId = ?
    AccountService->>DB: UPDATE account SET frozen_balance = frozen_balance + 15.75 WHERE userId = ?
    AccountService->>DB: INSERT INTO freeze_record (...)
    AccountService->>DB: COMMIT
    AccountService-->>TaskService: 冻结成功
    
    TaskService->>DB: INSERT INTO task (...)
    TaskService-->>User: 返回任务信息
```

#### 结算流程

```mermaid
sequenceDiagram
    participant Requester as 需求方
    participant TaskService as TaskService
    participant AccountService as AccountService
    participant DB as MySQL

    Requester->>TaskService: 验收通过
    TaskService->>AccountService: settle(taskId)
    
    AccountService->>DB: BEGIN TRANSACTION
    AccountService->>DB: UPDATE account SET frozen_balance = frozen_balance - 15.75 WHERE userId = 需求方ID
    AccountService->>DB: UPDATE account SET balance = balance + 15.00 WHERE userId = 接单方ID
    AccountService->>DB: UPDATE account SET balance = balance + 0.75 WHERE userId = 平台ID
    AccountService->>DB: INSERT INTO transaction_record (...)
    AccountService->>DB: COMMIT
    
    TaskService->>DB: UPDATE task SET status = 'COMPLETED'
    TaskService-->>Requester: 返回成功
```

### 4.5 WebSocket实时通知方案

#### 通知类型

| 类型 | 触发场景 | 通知对象 |
|------|----------|----------|
| TASK_ACCEPTED | 任务被接单 | 需求方 |
| TASK_SUBMITTED | 任务已提交 | 需求方 |
| TASK_APPROVED | 任务验收通过 | 接单方 |
| TASK_REJECTED | 任务验收拒绝 | 接单方 |
| DISPUTE_CREATED | 发起争议 | 双方 + 管理员 |
| DISPUTE_RESOLVED | 争议已处理 | 双方 |

#### WebSocket连接流程

```mermaid
sequenceDiagram

    Client->>Server: WebSocket连接 /ws/notifications?token=xxx
    Server->>Server: 验证Token，获取userId
    Server-->>Client: 连接成功

    loop 监听消息
        Server->>Client: WebSocket推送消息
    end
```

### 4.6 并发抢单方案（乐观锁/防超抢）

#### 抢单流程图

```mermaid
sequenceDiagram
    participant ClientA as 用户A
    participant ClientB as 用户B
    participant Server as 后端
    participant DB as MySQL

    ClientA->>Server: POST /tasks/1/accept
    ClientB->>Server: POST /tasks/1/accept

    Server->>DB: SELECT * FROM task WHERE id = 1 AND status = 'PENDING'
    Server->>DB: UPDATE task SET status = 'ONGOING', helper_id = A WHERE id = 1 AND version = currentVersion
    Server-->>ClientA: 接单成功
    Server-->>ClientB: 409冲突 - 任务已被接单
```

#### 乐观锁实现

```java
// 更新时校验版本号
@Update("UPDATE task SET status = #{status}, helper_id = #{helperId}, version = version + 1 WHERE id = #{id} AND version = #{version}")
int updateTaskWithVersion(Task task);
```

#### 失败响应示例

```json
{
  "code": 409,
  "message": "任务已被其他人接单",
  "data": {
    "currentStatus": "ONGOING",
    "conflictType": "STATUS_CHANGED"
  }
}
```

## 5. 开发规范

### 5.1 接口规范

| 项目 | 规范 |
|------|------|
| 基础路径 | `/api/v1/` |
| 命名风格 | 小写字母 + 连字符 |
| HTTP方法 | GET(查询)、POST(创建)、PUT(更新)、DELETE(删除) |
| 分页参数 | `pageNum`(默认1)、`pageSize`(默认10) |
| 成功响应 | `{ "code": 200, "message": "...", "data": {}, "timestamp": 1700000000000 }` |
| 失败响应 | `{ "code": 4xx, "message": "...", "data": null, "timestamp": 1700000000000 }` |

### 5.2 数据库命名规范

| 对象类型 | 规范 | 示例 |
|----------|------|------|
| 表名 | 小写 + 下划线 | `user`, `task`, `transaction_record` |
| 字段名 | 小写 + 下划线 | `user_id`, `task_title`, `created_at` |
| 主键 | 表名_id | `user_id`, `task_id` |
| 外键 | 关联表名_id | `requester_id`, `helper_id` |
| 索引 | idx_表名_字段名 | `idx_task_status`, `idx_user_phone` |

### 5.3 代码分层规范

| 层级 | 职责 | 命名规范 |
|------|------|----------|
| Controller | 接收请求、参数校验、返回响应 | 后缀Controller |
| Service | 业务逻辑处理 | 后缀Service |
| Mapper | 数据库CRUD操作 | 后缀Mapper |
| Entity | 数据库表映射 | 无后缀 |
| DTO | 数据传输对象 | 后缀Request/Response |
| Config | 配置类 | 后缀Config |
| Filter | 过滤器 | 后缀Filter |
| Util | 工具类 | 后缀Util |
| Enum | 枚举类 | 后缀Enum |

### 5.4 错误码规范

| 错误码范围 | 含义 |
|------------|------|
| 200 | 成功 |
| 400 | 请求参数错误 |
| 401 | 未登录或Token失效 |
| 403 | 无权限访问 |
| 404 | 资源不存在 |
| 409 | 资源冲突 |
| 422 | 业务逻辑错误 |
| 429 | 请求过于频繁 |
| 500 | 服务器内部错误 |
| 1001-1999 | 用户模块错误 |
| 2001-2999 | 任务模块错误 |
| 3001-3999 | 资金模块错误 |
| 4001-4999 | 系统模块错误 |

## 8. 总结

本技术方案针对校园互助众包平台的需求，采用：

1. **前后端分离架构**：Vue3 + Spring Boot，便于团队协作开发
2. **轻量级技术栈**：避免引入复杂中间件，降低学习和部署成本
3. **完善的安全机制**：JWT认证、RBAC权限、乐观锁防超抢
4. **事务一致性保障**：资金冻结、结算采用数据库事务
5. **实时通知能力**：WebSocket实现实时消息推送

适合2人团队快速开发，代码结构清晰，易于维护和扩展。
